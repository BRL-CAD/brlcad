  Long-Lived Application Containers

  These are the clearest targets for app-owned resources. They create a struct application, assign one resource, and delete the app later. Today cleanup is implicit via rt_i_destroy():

  - TclCAD command app: init src/libtclcad/commands.c:6560, delete src/libtclcad/commands.c:5196.
  - MGED command app with heap resource, likely leaking the resource container itself today: init src/mged/cmd.cpp:2271, delete src/mged/cmd.cpp:2190.
  - MGED WDB Tcl command app: init src/mged/wdb_obj.c:1382, delete src/mged/wdb_obj.c:1304.
  - NIRT owns long-lived app/rtip/resource pairs and resets/destroys them: src/libanalyze/nirt/nirt.cpp:453, src/libanalyze/nirt/nirt.cpp:468.

  These should move to “app owns resource; app destructor cleans with rt_clean_resource_basic() and frees storage if heap allocated.”

  rt_uniresource Replacement Sites

  The primary rt_uniresource blockers are fallback behavior, not unavoidable design needs:

  - rt_shootray() defaults missing a_resource to rt_uniresource: src/librt/shoot.c:603.
  - rt_shootray_bundle() does the same: src/librt/bundle.c:105.
  - rt_vshootray() does the same: src/librt/vshoot.c:140.
  - rt_cell_n_on_ray() has the same fallback: src/librt/shoot.c:1055.
  - Parallel prep still treats slot 0 / rt_uniresource as a special validity check: src/librt/prep.cpp:272.

  Direct users needing explicit local or app-owned resources if rt_uniresource goes away include beset, bigE, wireframe_eval, fortray, sh_wood, BOT sampling checks, arb8_tests,
  rt_boolweave, and the fuzz shooter.

  Conclusion

  The migration looks achievable. I do not see a feature requirement that forces rt_i cleanup discovery to remain. The required replacement is explicit ownership:

  - struct application should own one resource for single-thread/local app cases.
  - Parallel jobs should own resource arrays and hand each worker app its resource.
  - Long-lived Tcl/MGED/NIRT app containers should clean their resource in the app delete path.
  - Stats-producing render jobs must post stats before resource cleanup.
  - rt_shootray() and bundle/vshoot/cell helpers need to validate/init ap->a_resource without registering it in rtip->rti_resources.

  The first practical refactor target should be the long-lived application containers plus the shoot-path registration assertion. That removes the most important dependency on
  rti_resources discovery without disturbing the more complicated worker-array callers.
