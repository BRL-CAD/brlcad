== BRL-CAD Deep Inventory: Features and Algorithms

* *Inventory date:* April 2026
* *Source files:* ~7k total(~800 headers, ~1,500 C files, ~750 C{plus}{plus} files)

BRL-CAD is a cross-platform, open-source graphics system with origins in
the U.S. Army Ballistic Research Laboratory. It has been under active
development since 1979 and is the primary tri-service solid modeling CAD
system used for vehicle vulnerability and lethality analyses. The
codebase exceeds one million lines of source code spanning geometry
modeling, ray tracing, image/signal processing, analysis, and format
conversion.

=== 1. Solid Geometry Representation

==== 1.1 Constructive Solid Geometry (CSG)

BRL-CAD’s primary geometric paradigm is CSG. Objects are arranged in a
hierarchical tree of _combinations_, where leaf nodes are named solid
primitives and internal nodes are boolean operators. Three boolean
set-theoretic operations are supported: *union*, *intersection*, and
*difference (subtraction)*. Combinations themselves can be used as
operands, producing deeply nested expression trees. Each arc in the tree
carries a 4×4 homogeneous transformation matrix that positions or scales
the primitive in model space. Any combination can be declared a
_region_, which is the smallest unit carrying material identity (e.g.,
material ID, region code, item code). Regions are the fundamental unit
for analysis such as volume, mass, and overlap detection.

A complete CSG ray-tracing engine with per-primitive analytical
intersection for 30{plus} primitive types, parallel multi-core
execution, a choice of BSP-tree or HLBVH spatial acceleration, and
boolean segment weaving across arbitrarily deep boolean trees.

==== 1.2 Implicit CSG Primitives

The following analytically-defined (implicit) solid primitives are
supported. Each carries a dedicated ray-solid intersection algorithm, a
surface-normal computation, UV mapping, surface curvature computation,
volume formula, surface area formula, centroid formula, tessellation to
NMG mesh, and conversion to B-rep (NURBS):

* *TOR* – Torus (ring torus defined by center, normal axis, major
radius, minor radius)
* *TGC* – Truncated General Cone (generalized cone with elliptical
cross-sections at each end; subsumes right circular cylinder REC, right
circular cone, and other degenerate forms)
* *ELL / SPH* – Ellipsoid (general tri-axial) and Sphere (degenerate
case)
* *SUPERELL* – Superellipsoid (exponent-parameterized ellipsoid
supporting squircle-like shapes)
* *ARB8* – Arbitrary convex polyhedron defined by up to eight vertices
(subsumes ARB7, ARB6, ARB5, ARB4 with degenerate/shared vertices);
supports up to 5-sided planar faces
* *ARBN* – Arbitrary convex solid defined by N half-space plane
equations (the intersection of N half-spaces); supports any convex solid
with planar faces
* *ARS* – Arbitrary Regular Solid: surface of revolution defined as a
sequence of 2D cross- section curves (each a closed polygon); the solid
is the surface swept between them
* *HALF* – Half-space: all points on one side of an infinite plane
(defined by a single plane equation); useful as a cutting solid in
boolean subtraction
* *EBM* – Extruded Bit Map: a binary raster image (1-bit per pixel)
extruded to a specified height, generating a 3D solid from a 2D
black-and-white image or BRL-CAD object
* *VOL* – 3D Voxel Volume: a three-dimensional array of voxels where a
threshold range selects which cells are considered solid; enables CT/MRI
scan geometry representation
* *HF* – Height Field: a 2D grid of elevation samples forming a terrain
surface, with configurable cell scale and Z-scaling; supports adaptive
triangle-cut selection
* *DSP* – Displacement Surface/Map: similar to HF but stored as an
embedded database object or file, with smooth-surface normal
interpolation and adaptive triangle-cut direction control
* *PIPE* – Pipe solid with variable inner diameter, outer diameter, and
bend radius along a piecewise-linear spine; models pipes, tubes, and
wire with optional hollow cores
* *PARTICLE* – Lenticular particle solid: a rounded cylinder (capsule)
defined by two endpoint radii; degenerates to a sphere or cylinder; used
for blast fragment modeling
* *RPC* – Right Parabolic Cylinder: parabolic cross-section extruded;
one of the conic-section extruded primitives
* *RHC* – Right Hyperbolic Cylinder: hyperbolic cross-section extruded
* *EPA* – Elliptical Paraboloid: paraboloid with elliptical
cross-sections
* *EHY* – Elliptical Hyperboloid: hyperboloid of one sheet with
elliptical cross-sections
* *HYP* – Hyperboloid of two sheets with elliptical cross-sections;
parameterized by neck width ratio for modeling hyperboloid cooling-tower
shapes
* *ETO* – Elliptical Torus: a torus swept from an ellipse rather than a
circle; generalizes the standard torus
* *HRT* – Heart: a smooth, curved heart shape defined by a center,
directional axes, and cusp distance parameter
* *CLINE* – FASTGEN CLINE element: a finite-length cylindrical rod with
a specified radius and wall thickness, used for structural rods in
military vehicle FASTGEN models; supports both solid and hollow (tube)
modes
* *METABALL* – Metaball / Blobby Surface: an implicit isosurface defined
by a set of weighted control points; three evaluation methods are
supported:
** _Isopotential_ (inverse-square field strength)
** _Blinn Blobby_ (exponential field)
** _Tokyo Metaball_ approximation
* *EXTRUDE* – Profile Extrusion: a 2D sketch profile (see below)
extruded along a vector, producing a prismatic solid; the extrusion
vector need not be perpendicular to the sketch plane
* *REVOLVE* – Surface of Revolution: a 2D sketch profile revolved around
an axis by a specified angle, producing a solid of revolution; partial
revolutions are supported

==== 1.3 Explicit Geometry Representations

* *BoT (Bag of Triangles)* – Triangle mesh with four operating modes:
** _Surface mode_: triangles represent a 2D surface (no enclosed volume)
** _Solid mode_: closed triangle mesh representing a solid volume;
orientation-consistent
** _Plate mode_ (FASTGEN): each triangle has an associated thickness;
used for thin-walled structures in military FASTGEN models; plate normal
direction is face-center-offset
** _Plate-NoCoS_: plate mode variant where LOS equals face thickness
rather than thickness divided by cosine of obliquity angle
** Supports per-vertex surface normals, per-face UV texture coordinates,
and a float-precision mode to reduce memory for very large meshes;
stores face normals as index arrays
** HLBVH (Hierarchical Linear Bounding Volume Hierarchy) acceleration
for large meshes
** Oriented bounding box (OBB) computation for BoT primitives
** CSG-to-BoT Boolean tessellation pipeline
* *NMG (Non-Manifold Geometry)* – Weiler radial-edge data structure (see
Section 6) for topologically general polyhedral solids; supports wire
edges, lone vertices, and non-manifold configurations
* *BREP (Boundary Representation)* – NURBS-based B-rep using the
openNURBS library; supports trimmed NURBS surfaces,
complex edge topology, and all B-rep operations (see Section 5)
* *SKETCH* – 2D profile consisting of an arbitrary mix of line segments,
circular arcs, NURB curve segments, and Bézier curve segments defined in
a local 2D coordinate system; used as input to EXTRUDE and REVOLVE, and
standalone as an annotation aid
* *PNTS* – Point cloud: an unordered set of 3D points with optional
per-point attributes (color, scale, surface normal), memory-optimized by
storing only the attributes needed
* *BSPLINE (legacy NURBS)* – Legacy NURBS surface storage; newer code
uses BREP
* *PG (Polygon)* – Legacy explicit polygon solid (multi-polygon faces,
each with vertex normals); largely superseded by BoT

==== 1.4 Non-Geometric / Structural Objects

* *COMBINATION* – Boolean CSG tree node referencing other named objects;
carries material assignment, color, and region-flag
* *SUBMODEL* – Embeds another .g file (or sub-tree of the same file) as
a reusable component that is ray-traced independently with its own space
partitioning
* *GRIP* – Display-only grip/handle object used by animation scripts
* *JOINT* – Kinematic joint object connecting two geometry paths; used
in animation to define articulated assemblies
* *DATUM* – Reference geometry entity per ASME Y14.5M: stores points,
lines, and planes as coordinate reference features; supports linked
lists of multiple datum elements
* *ANNOT (Annotation)* – 2D in-scene text label with leader line;
references a sketch for geometry; supports nine anchor positions
relative to a reference point
* *SCRIPT* – Script-type object with a type string for future automation
hooks
* *BINUNIF* – Binary uniform data (generic typed data arrays embedded in
the database)

=== 2. Ray-Solid Intersection Algorithms

==== 2.1 Per-Primitive Intersection

Every primitive type implements a function that computes parametric
ray-entry/exit distances. Techniques used include:

* *Quadric surface intersection* for ELL, TOR, SPH, TGC, RPC, RHC, EPA,
EHY, HYP, ETO, HRT: substituting the ray parametric equation into the
implicit algebraic equation produces a polynomial in the ray parameter.
Polynomial roots (quadratic, cubic, quartic) are found analytically
using closed-form formulas in `libbn`. Coefficients are pre-computed
during prep from the primitive parameters.
* *Half-space intersection* for HALF, ARB8, ARBN: ray vs. infinite
plane, intersect and clip between pairs of entry/exit half-spaces.
* *Height field / DSP intersection*: the ray is marched through a 2D
grid of triangulated cells. Each height sample pair defines two right
triangles; the ray is tested against each triangle the ray footprint
covers. Adaptive triangle-cut direction (upper-right / lower-left, or
adaptive) minimizes artifacts.
* *EBM intersection*: marches the ray through the 2D bitmap cell grid,
testing solid/empty voxel transitions using the extruded height as the Z
extent.
* *VOL intersection*: steps through a 3D voxel grid, testing threshold
conditions.
* *BoT intersection* via Möller–Trumbore ray-triangle intersection
algorithm; large meshes use HLBVH (see Section 2.3) for sub-solid
acceleration; a special single-precision float variant reduces memory
for massive meshes.
* *PIPE intersection*: decomposes the pipe into bent tube segments; each
segment is handled as a truncated cone or sphere joining adjacent linear
segments, with iterative numerical root finding when necessary.
* *METABALL intersection*: marches along the ray at decreasing step
sizes to find the isosurface crossing; `initstep` and `finalstep`
parameters control step sizes.
* *BREP intersection*: delegates to the openNURBS ray-NURBS intersection
engine (Newton iteration in parameter space).
* *EXTRUDE / REVOLVE*: intersection derived analytically from the 2D
sketch geometry extruded or revolved.
* *SKETCH*: not directly ray-traced; used to generate EXTRUDE/REVOLVE
solids.
* *SUBMODEL*: fires secondary ray-tracing instance into the referenced
sub-tree.
* *ARS*: the surface is tessellated into triangles for intersection.
* *SUPERELL*: quartic polynomial intersection via analytical root
finding.

Each primitive also implements:

* surface normal at a hit point
* UV texture coordinates at a hit point
* principal curvature computation at a hit point
* axis-aligned bounding box computation
* volume, surface area, centroid physical property formulae
* tessellation to NMG mesh
* conversion to openNURBS B-rep
* wireframe plot and LoD wireframe generation

==== 2.2 Boolean Evaluation (Boolweave)

After each solid along a ray reports its in/out intersection segments,
boolean weaving merges all segments into a sorted partition list (one
linked-list entry per distinct depth interval). Finally, the CSG
boolean expression tree is evaluated against this partition list
(walking the binary boolean tree to determine which partitions are
"`in`" the final solid) and calls the application’s hit` callback for
each valid surface hit. Edge cases handled include zero-thickness
segments (wire edges, isolated vertices) and numerical tolerance
handling for near-coincident surfaces.

==== 2.3 Space Partitioning

Two space partitioning strategies co-exist:

*BSP (Binary Space Partition) tree with axis-aligned splitting planes:

* Builds a binary kD-tree by recursively splitting the model bounding
box along X, Y, Z axes
* Split position measures the expected cost of a split (weighted by
solid count in each child), iterating through candidate positions
* post-processes the tree to
eliminate cells that are more than 50% empty by splitting at the
geometry’s actual boundary within the cell
* Tree construction is parallelized with a work queue of subtrees
* Supports dynamic insertion/removal of individual solids

*HLBVH (Hierarchical Linear Bounding Volume Hierarchy)*:

* Derived from the pbrt-v3 HLBVH algorithm by Garanzha, Pantaleoni, and
McAllister (HPG 2011)
* Computes Morton codes for primitive centroids by encoding 3D
position into a 30-bit integer using left-shift bit-interleaving
(Z-order / Morton curve)
* Radix sorts primitives by Morton code (6 bits per pass, 5 passes)
* Builds LBVH treelets from Morton-ordered clusters using a fixed-bit
splitting strategy
* Upper tree levels use SAH (Surface Area Heuristic) for optimal
splitting
* Used as an alternative to the BSP tree; available for OpenCL
acceleration path

*Infinite solid handling*: solids with no bounding box (e.g., HALF)
are collected and tested on every ray after the finite BSP tree is
exhausted.

*Large-solid piece decomposition*: some large primitives (particularly
BoT) decompose into multiple spatial "`pieces,`" each with its own
bounding box, allowing the space partition tree to skip pieces whose
bounding boxes the ray misses.

==== 2.4 Ray Bundle Shooting

A set of pattern generators can produce structured ray bundles:

* Rectangular orthographic and perspective grids
* Circular orthographic and perspective grids
* Circular spiral patterns
* Elliptical orthographic/perspective grids
* Cylindrical layer patterns
* Spherical layer patterns
* Quasi-random unbiased spherical volume sampling

==== 2.5 Parallel SMP Ray Tracing

The ray tracer is designed for symmetric multi-processing.  Dispatches
N worker threads, each with a private `struct resource` containing
per-thread memory pools for hit records, partitions, segments, and
free lists (avoiding lock contention). Global semaphores protect only
shared structures (work assignment, framebuffer output). The thread
count is configurable; recursive parallel calls are supported up to a
64×64 thread limit.

=== 3. NURBS Boundary Representation (B-rep)

BRL-CAD incorporates a full B-rep kernel via the openNURBS library (from
McNeel/Rhino), wrapped and extended with application-specific
algorithms.

Full NURBS B-rep support including surface-surface intersection via
Newton iteration in parameter space, B-rep boolean evaluation
(union/intersection/difference), Constrained Delaunay Triangulation for
visualization, and pullback curve computation.

==== 3.1 Core B-rep Storage

B-rep geometry is stored as OpenNURBS objects, providing for trimmed
NURBS surface representation.  Each face is a NURBS surface (rational
or non-rational, any degree, with arbitrary knot vectors) trimmed by
2D parameter-space boundary curves. Edges carry 3D curves. The
complete topological structure (vertices, edges, edge curves, face
surfaces, trims, loops) follows the openNURBS spec.

==== 3.2 Surface-Surface Intersection (SSI)

The `intersect.cpp` module implements NURBS surface-surface intersection
for B-rep boolean evaluation. The algorithm:

1. Subdivides surfaces using a Bézier clipping / knot insertion approach
to isolate intersection regions
2. Applies Newton-Raphson iteration in the 4D parameter space
(u1,v1,u2,v2) to converge to intersection curve points
3. Traces intersection curves by marching with adaptive step size
4. Handles tangential intersections and closed intersection loops

==== 3.3 B-rep Boolean Operations

Evaluates union, intersection, and difference Brep objects:

1. Computes surface-surface intersection curves between face pairs
2. Classifies resulting face portions as "`inside`" or "`outside`" the
opposing solid using point-membership tests (ray casting from a face
point into the other solid)
3. Splits trimming loops at intersection curve endpoints, creating new
trimmed face regions
4. Reconnects the surviving half-faces into a valid closed B-rep topology
5. Intersection curves are represented as `LinkedCurve` chains linking
`SSICurve` segments that share endpoints; directed graph connectivity
determines trim loop assembly

==== 3.4 Pullback Curve Computation

`PullbackCurve.cpp` computes the 2D parameter-space representation of a
3D intersection curve on a NURBS surface. Uses Newton iteration
projected onto the surface, maintaining continuity with a
predictor-corrector scheme.

==== 3.5 Constrained Delaunay Triangulation (CDT) for B-rep

`libbrep/cdt/` implements a full CDT pipeline for B-rep visualization
and analysis:

* Triangulates individual trimmed NURBS faces into flat triangle meshes
* Enforces face trim loop edges as Delaunay constraints
* Handles mesh topology at shared edges between adjacent faces; resolves
vertex ordering conflicts at shared edges
* Produces watertight triangle meshes from closed B-rep objects
* Detects and reports mesh topology errors: unmatched edges, misoriented
triangles, degenerate faces

==== 3.6 B-rep Repair

`edit.cpp` in libbrep supports repair of topologically invalid B-rep
objects:

* Detection and repair of bad 3D edge curves using Loft approximation
from 2D parametric data
* Identification of invalid trim loops and degenerate faces

==== 3.7 Shape Recognition

Shape recognition identifies geometric shape types (cylinder, cone,
sphere, torus, planar surfaces, etc.) in B-rep face groupings,
enabling approximate conversion from B-rep to CSG primitives.

==== 3.8 Subcurve and Subsurface

Subcurve and subsurface bounding-box tree hierarchies are evaluated
over NURBS curve/surface parameter domains, implementing fast spatial
lookup for intersection candidate pairs.

=== 4. Non-Manifold Geometry (NMG)

==== 4.1 Data Structure

A complete Non-Manifold Geometry (NMG) kernel using Weiler’s radial-edge
data structure, supporting fully general non-manifold topology with
NMG-level boolean operations. NMG uses Kevin Weiler’s radial-edge data
structure (from "`Non-Manifold Geometric Boundary Modeling,`" SIGGraph
1989), extended and implemented for BRL-CAD by Michael Muuss and Lee
Butler.

The hierarchy is:

*Model → Region → Shell → Face/Loop → Edge/Vertex*

* *Model*: root container
* *Region*: a set of shells (one connected body per shell)
* *Shell*: a closed or open manifold surface
* *Face* with a *faceuse* pair (two half-faces, one per orientation)
* *Loop* (outer {plus} inner loops defining the face boundary)
* *Edge* with two *edgeuse* entries (one per adjacent face); the
_radial_ pointer links all edgeuses around the same geometric edge,
enabling non-manifold connectivity
* *Vertex* shared by all incident edges

==== 4.2 NMG Operations

* Topological validity checking
* Boolean operations on NMG solids (union, intersection, difference)
using face-face intersection and the radial-edge topology to track
in/out classification
* Tessellation of NMG surfaces into flat triangular faces
* Conversion to BoT mesh
* Ray intersection with NMG solids (via face-plane and loop-boundary
tests)
* NURBS conversion (each planar face becomes a degenerate NURBS face)
* Face plane normal flipping, loop reversal
* Vertex fusion (merging geometrically coincident vertices within
tolerance)
* NMG model I/O (reading and writing)
* Visualization (wireframe plot of edges)
* Walking algorithms (visiting all model elements)
* Index remapping for serialization

=== 5. Ray Tracing and Rendering

==== 5.1 Primary SMP Ray Tracer

The core ray tracer (`rt`) produces high-quality rendered images:

* Shoots primary rays through each pixel (parallel, using the SMP
infrastructure)
* Calls application hit/miss callbacks; the default renders using the
optical shader assigned to each region
* Supports multi-frame animation sequences via the animation system
* Configurable: resolution, antialiasing, number of CPUs, background
color, ambient light, perspective, hyper-sampling
* Output formats: raw pix (RGB), BW, PNG, and others via libicv
* Benchmarking mode produces reference images for performance comparison

==== 5.2 ADRT (Advanced Distributed Ray Tracer)

Path tracing with Monte Carlo global illumination via the ADRT renderer:

* *Path tracing* (`librender/path.c`): Monte Carlo path tracing with
global illumination, supporting diffuse inter-reflection
* *Phong shading* (`librender/phong.c`): classical Phong BRDF with
ambient, diffuse, and specular components
* *Depth renderer* (`librender/depth.c`): encodes ray depth as grayscale
* *Normal renderer* (`librender/normal.c`): surface normal visualization
* *Component renderer* (`librender/component.c`): highlights individual
geometry components
* *Hit renderer* (`librender/hit.c`): single-hit visualization
* *Grid renderer* (`librender/grid.c`): grid overlay visualization
* *Flat renderer* (`librender/flat.c`): unshaded flat color per region
* *Surfel renderer* (`librender/surfel.c`): surfel-based rendering
* *SPALL renderer* (`librender/spall.c`): spall fragment visualization
* *FLOS renderer* (`librender/flos.c`): fragment line-of-sight
visualization
* *Texture system*: camo, checker, blend, clouds, gradient, image, mix,
perlin noise, bump map, stack (layered texture composition)
* Master/slave distributed rendering architecture (`master/`, `slave/`)
for network- distributed ray tracing across multiple machines
* Uses Tcl scripting for session control

==== 5.3 ART (Application Ray Tracer)

`art` provides an alternative rendering front-end with saved-view
support and configurable rendering quality settings.

==== 5.4 ISST (Interacting Solid Solids Tracer)

`isst` is an interactive solid tracer used for real-time display of CSG
geometry via the ADRT rendering engine, with a Tcl/Tk GUI.

==== 5.5 Remote Distributed Ray Tracing

`remrt` implements a network client-server architecture for distributing
ray tracing work across multiple machines over TCP/IP, using `libpkg`
for packet-based communication.

==== 5.6 OpenCL Acceleration

Selected primitives have OpenCL kernels: when compiled with OpenCL
support, BoT ray-triangle intersection and the boolean weave step can
offload to GPU. The HLBVH structure is converted to a linear flat array
suitable for GPU traversal.

=== 6. Optical / Material Shading Models

The optical library (`liboptical`) implements a shader plugin system.
Shaders are assigned per-region and evaluated at ray-surface
intersection points to determine color and light contribution. All
shaders expose a shared interface (init, setup, render, free).

* *Plastic*: classical Phong reflection model with
ambient, diffuse, and specular terms; supports mirror reflections via
recursive ray casting
* *Mirror* (degenerate plastic): perfect specular reflection
* *Glass* (degenerate plastic): perfect specular transmission with
Snell’s law refraction
* *Phong BRDF*: physically-based Phong with explicit
BRDF formulation
* *Cook-Torrance*: microfacet specular reflectance
using the Beckmann distribution for physically-based metallic and
dielectric materials
* *Stack*: allows layering of multiple shaders whose
results are composited
* *Light*: area and point light source shaders;
support for colored lights, shadow testing, and soft shadows via
distributed sampling
* *Null*: transparent shader (invisible geometry)
* *Air*: atmospheric medium with configurable
density/transmittance; supports exponential falloff with height
* *Cloud*: volumetric cloud
appearance
* *Fire*: procedural fire appearance
* *Fog* / *Smoke*: density-based atmospheric scattering
* *Grass*: procedural grass surface
* *Wood*: procedural wood grain texture
* *Camouflage*: multi-scale procedural camouflage
pattern using fractal noise
* *Texture mapping*:
reads image files and applies them as surface textures; supports
spherical and planar projection
* *Billboard*: billboard sprite texture applied
to geometry
* *Noise shaders*:
various noise-based surface perturbations
* *Points*: point-sampled surface appearance
* *Bump map*: perturbs surface normals using a
texture map
* *SPM (Spherical projection map)*: spherical
environment map lookup
* *Toon*: cel-shaded cartoon rendering
* *Toyota*: automotive paint shader with metallic
flake modeling
* *Treetherm*: temperature-based colorization for
thermal analysis
* *Flat*: constant flat color
* *Extern / TCL*: Tcl-scriptable shader for custom user
shading functions
* *OSL (Open Shading Language)*: OSL shader
integration via liboslrend, enabling scripted physically-based materials

==== 6.1 Photon Mapping

A photon mapping global illumination renderer with kD-tree photon
storage, nearest- neighbor gather, irradiance caching, and support for
caustic and indirect light maps.

`photonmap.c` implements a photon mapping global illumination algorithm:

* Photon emission from light sources (Monte Carlo sampling of emission
directions)
* Photon transport via recursive ray tracing with Russian roulette for
path termination
* Photon storage in a *kD-tree* (built by median-split along the widest
bounding-box axis)
* Photon gathering using a *nearest-neighbor search* in the kD-tree with
a maximum search radius (spherical query)
* Irradiance estimation at hit points by summing gathered photon
contributions weighted by a cone filter
* *Irradiance caching* (`IC`): caches computed irradiance values on a
grid to avoid re-computation; hypersampling at cache misses
* Separate photon maps for: direct illumination, indirect illumination,
specular/caustic photons, and volume caustics
* Multi-threaded photon tracing using the SMP resource system

==== 6.2 Refraction and Transparency

`refract.c` implements Snell’s law refraction and the Fresnel equations
for dielectric interfaces; used by the glass/plastic shader for
physically correct transparency.

==== 6.3 Spectral Data Support

`libbn`’s tabular data system
provides wavelength-indexed spectral curve storage and arithmetic (add,
multiply, integrate, convolve). This is used by the optical library to
represent spectral power distributions, albedo, and reflectance curves
for physically-based spectral rendering.

=== 7. Geometric Analysis

==== 7.1 Geometry Quality Analysis

A comprehensive ray-based geometric analysis framework for volume, mass,
centroid, moments of inertia, surface area, overlap detection, gap
analysis, and voxelization — all derived from structured-grid ray
casting with configurable adaptive refinement.

A comprehensive geometry quality analyzer (`gqa`) fires grids of rays at
user-specified objects to measure:

* *Volume*: computed by integrating ray segment lengths over a grid of
parallel rays; a three-direction cross-fire strategy improves accuracy;
error bounds estimated from spacing
* *Mass*: volume × material density (looked up from a density database)
* *Centroid*: mass-weighted first moment via ray integration
* *Moments of inertia*: mass-weighted second-moment tensor via ray
integration
* *Surface area*: via the `libanalyze` surface area API
* *Overlaps*: detects regions of space where two different CSG regions
coexist (interpenetrate), reporting maximum depth and coordinates; uses
per-thread overlap tables and semaphores
* *Gaps*: detects unmodeled empty space between adjacent regions that
should be adjacent
* *Adjacent air*: detects air regions directly adjacent to material
regions without an intervening interface
* *Exposed air*: air regions accessible from outside the model
* *Unconfined air*: air regions that are not enclosed by material
* Grid spacing is adaptively refined until convergence criteria are met

==== 7.2 Ray-Based Volume and Mass Analysis (libanalyze API)

`libanalyze` exposes an API for:

* volume per object or entire model
* mass integration
* centroid computation
* inertia tensor
* surface area

All are driven by ray tracing which fires a configurable grid from
multiple azimuth/elevation directions, with configurable grid spacing
and refinement limit.

==== 7.3 NIRT (Natalie’s Interactive Ray Tracer)

`libanalyze/nirt/` and the standalone `nirt` tool fire individual named
rays at geometry and produce detailed intersection reports along the
ray:

* Region name, path, solid name at each hit
* Hit point coordinates and surface normal
* Region entry/exit distances, line-of-sight (LOS) thickness
* Overlap reports with depth at each overlap
* Gap reports with gap distance
* Air region reports
* Configurable output format strings (printf-style with named fields)
* Multiple output format presets
* Overlap handling modes: resolve (merge), rebuild FASTGEN, rebuild all,
retain
* Callback hooks for programmatic use (segment callbacks, gap callbacks,
etc.)
* Plot output with color-coded segment types

==== 7.4 Voxelization

Convert CSG geometry to a 3D voxel grid by:

* Firing three orthogonal sets of rays through the model
* Accumulating region occupancy and fill fraction per voxel
* Calling a user callback for each occupied voxel with its index and
fill fraction
* Level-of-detail parameter controls sub-voxel sampling

==== 7.5 Implicit Surface Polygonizer (Marching Cubes / Tetrahedral)

An implicit surface polygonizer (Bloomenthal tetrahedral marching) for
converting any inside/outside function to a triangle mesh.

`polygonizer.c` (based on Jules Bloomenthal’s "`An Implicit Surface
Polygonizer,`" Graphics Gems IV, 1994) converts any implicit function
(inside/outside test) to a triangle mesh:

* Maintains a hash table of evaluated lattice corners and edges
* Expanding front: starts at a seed point and spreads outward, testing
adjacent cube faces
* Each transitive cube is optionally decomposed into 6 tetrahedra for
sub-cube accuracy
* Edge-surface crossing is found by bisection (`converge`) then the
vertex normal is estimated from finite differences
* Used in `libanalyze` for converting CSG regions to faceted form

==== 7.6 Geometry Differencing (raydiff)

`raydiff.c` measures geometric differences between two versions of a
model by:

* Firing rays through both and comparing the partition lists
* Reporting regions where geometry has been added, removed, or
repositioned
* Produces difference reports useful for tracking model changes

==== 7.7 Geometry Checker (gchecker)

Analytical closed-form root finding for quadratic, cubic, and quartic

=== 13. Geometry Database System

==== 13.1 Database Architecture

The BRL-CAD geometry database (`.g` files) is a record-oriented binary
database with:

* A free list for deleted records
* Object directory stored as an in-memory hash table keyed by object

(variable-length records with external type/size prefix)
* All I/O mediated through a database instance structure
* Concurrent read access; single-writer serialization via semaphores
* Memory-mapped I/O for efficient large database access
* Object-level corruption detection via checksum and magic-number
validation

* Material properties (density, thermal conductivity, etc.)

attribute names with associated types for interoperability
* Attribute inheritance through combination trees
* Attribute synchronization during material assignment

==== 13.3 Database Diff and Version Control

* `gdiff` tool: compares two versions of a database

per-attribute change detail
* `reduce++_++db`: removes unused/unreferenced objects and
simplifies database

==== 13.4 Database Search and Query

`librt/search.cpp` provides a rich object-search interface:

* Filter by object name (glob or regex)
* Filter by object type (solid kind, region flag, combination flag)

* Filter by boolean relationship in tree
* Filter by primitive parameters (bounding box overlap, etc.)
* Recursive tree walking with composable predicates

==== 13.5 Animation System

==== 13.6 Object Naming and Paths

=== 16. Display, View, and Visualization (libdm / libbv)

==== 16.1 Display Manager (libdm)

Abstracts hardware/software rendering backends:

* OpenGL display manager for hardware-accelerated display
* Software (swrast) fallback
* Null/dummy display for headless operation
* Platform display managers (X11, Win32, macOS)
* Display list rendering using OpenGL VBOs
* Font rendering (`vfont.c`, BRL-CAD vector font)

==== 16.2 View Abstraction (libbv)

`bview` structure encapsulates all view state:

* Camera position, orientation, field of view, perspective
* Model-to-view and view-to-model transform matrices
* Screen dimensions
* LOD (level-of-detail) settings for adaptive display
* Scene object list

==== 16.3 Plot3 Format

BRL-CAD’s internal vector graphics format:

* 2D/3D line segments, polygons, points in integer and floating-point
variants
* Color operations
* Used extensively for debug output, wireframe visualization, and
ground-truth overlays

==== 16.4 Adaptive Level-of-Detail

Callbacks generate view-dependent wireframe representations:

* Curve tessellation density scales with view resolution (view size /
distance to object)
* BoT primitives cache per-LOD wireframe representations
* Draw command (`libged`) asynchronous population being explored

=== 17. Procedural and Parametric Geometry Generation

==== 17.1 Parametric Architectural Shapes

Procedural generators for common manufactured forms:

* *Bolt*: hex-head bolt with configurable diameter, length,
head dimensions
* *Coil*: helical coil spring with configurable pitch,
inner/outer diameter, number of turns, end types
* *Fence*: chain-link or picket fence with posts, rails, and
mesh geometry
* *Gastank*: propane/gas tank with cylindrical body and
hemispherical caps
* *Handle*: symmetric handle shape
* *Human*: simplified anthropomorphic human figure from
ellipsoids and cylinders, with configurable height and proportions
* *Tire*: automotive tire with tread geometry and sidewalls
* *Window / Window Frame*:
architectural window primitives
* *Wire*: wire segment primitives

==== 17.2 Procedural Database Geometry (proc-db)

Programs for programmatic model generation:

* *Molecular structures*: reads Protein Data
Bank files and creates sphere-and-stick molecular models; caffeine and
3NN4 example models included
* *Masonry*: brick and stone wall pattern generation
* *Building*: multi-story building
generation from floor plans
* *Metaballs*: metaball cluster
generation
* *Sphflake*: fractal sphere-cluster tree (similar to
Menger sponge principle but with sphere hierarchies)
* *Menger sponge*: recursive cube-based fractal geometry
* *Ringworld*: concentric ring terrain geometry
* *Globe*: spherical globe with geographic layers
* *Lens*: optical lens cross-section geometry
* *Clutter*: random-placed object fields for performance
testing
* *B-rep primitives*:
example B-rep model generation
* *NMG model*: programmatic NMG solid construction
example
* *Sketch*: 2D sketch construction example
* *CSG from B-rep*: B-rep to CSG decomposition example
* *Pipe*: pipe assembly generation
* *Brickwall*: stacked brick wall geometry
* *Pyramid*: pyramid with configurable sides
* *Gtimes*: geometry timing/performance test

=== 18. Performance Benchmark Suite

BRL-CAD has a standardized ray-tracing benchmark:

* Renders a set of fixed reference scenes at fixed resolution
* Compares output pixel values against stored reference images
(correct/incorrect)
* Reports performance in *VGR units* (relative to a DEC VAX 11/780
baseline)
* Also reports *rays/second* for absolute throughput
* Integrated into the CMake build system (`make benchmark`)
* Benchmark scenes test a range of primitive types, combination
complexity, and CSG depth
* Used for regression testing, platform comparison, and hardware
evaluation

=== 19. Scripting, Automation, and GUI Infrastructure

==== 19.1 Tcl/Tk Scripting

* *libtclcad*: library that exposes librt, libged, and display manager
functionality as Tcl commands; provides the scripting interface for all
BRL-CAD Tcl-based applications
* *bwish*: BRL-CAD’s Tcl/Tk interpreter with all BRL-CAD commands built
in
* *tclscripts/*: extensive library of Tcl scripts implementing the
classic MGED GUI, geometry wizards, analysis reporting, animation
scripting, and model management utilities

==== 19.2 MGED (Multiple Geometry EDitor)

`mged`: the classic interactive geometry editor and environment;
provides:

* Full command-line interface for all libged operations
* Interactive 3D wireframe view with OpenGL display
* Mouse-driven solid and combination editing with visual feedback
* Illumination of selected solids
* View control (rotate, translate, zoom, perspective)
* Overlap visualization, gap visualization
* A Tcl/Tk GUI layer on top of the display manager

==== 19.3 Archer (Tk-based GUI) / QGED (Qt-based GUI)

`archer` and `qged`: newer application GUIs providing:

* Hierarchical tree view of the database
* Embedded 3D view with libdm rendering
* Modern widget-based command interface

==== 19.4 RTWizard

`rtwizard`: a Tcl/Tk wizard interface that simplifies production
rendering from BRL-CAD geometry; configures ray tracer settings, view,
lighting, and output format.

=== 20. External Library Integrations

BRL-CAD bundles and integrates several third-party libraries:

* *openNURBS* (McNeel & Associates): NURBS surface representation and
B-rep kernel; central to all BREP primitive and STEP/IGES/3DM operations
* *Assimp* (Open Asset Importer Library): multi-format 3D model import
* *OpenVDB* (DreamWorks Animation): sparse volumetric data structure
* *GDAL* (Geospatial Data Abstraction Library): terrain and raster data
import
* *STEPcode*: STEP file parsing
* *Clipper* (Angus Johnson): 2D polygon boolean operations
* *QuickHull* (Antti Kuukka): 3D convex hull
* *nanoflann*: fast approximate nearest-neighbor search (used in SPSR)
* *Detria*: constrained Delaunay triangulation
* *earcut.hpp*: polygon ear-clipping triangulation
* *delaunator.hpp*: fast 2D Delaunay triangulation
* *OSL (Open Shading Language)*: physically-based shader language
runtime
* *MT19937*: Mersenne Twister PRNG
* *pbrt-v3* (derived): HLBVH construction algorithm
* *Bloomenthal polygonizer*: implicit surface polygonization
* *Screened Poisson Reconstruction* (Michael Kazhdan): surface
reconstruction from point clouds
* *Ball Pivoting Algorithm*: surface reconstruction
* *Creo/Pro-E, Unigraphics/NX, Cubit*: external CAD system geometry
export interfaces
