Name

ch_open - create and manipulate a command history object.

SYNOPSIS

ch_open[chName]

DESCRIPTION

The ch_open command creates a new command history object (given by the chName argument). The ch_open command returns its chName argument. At the time this command is invoked, there must not exist a command history object named chName.

METHODS

The ch_open command creates a new Tcl command whose name is chName. This command may be used to invoke various operations on the command history object. It has the following general form:

chName option ?arg arg …​?

Option and the args determine the exact behavior of the command. The following commands are possible for command history objects:

chName add command_string chName *addM command_string*

Add a command to the history and reset the current command pointer.

chName curr

Return the current command in the history.

chName history

Return the entire history.

chName next

Return the next command in the history making that the current command.

chName prev

Return the previous command in the history making that the current command.

EXAMPLE

The following sequence creates a command history object, adds commands and traverses the history.

cwish> ch_open ch
ch
cwish> ch add "cmd1"
cwish> ch add "cmd2"
cwish> ch prev
cmd2
cwish> ch prev
cmd1
cwish> ch curr
cmd1
cwish> ch add "cmd3"
cwish> ch prev
cmd3
cwish> ch prev
cmd2
cwish> ch next
cmd3
cwish> ch next
cwish> ch history
cmd1
cmd2
cmd3

Compact command schemas

The bu_cmd_schema API in bu/cmdschema.h describes the common, flat command shape in one compact, code-native table. It is intended for commands whose option spelling, type, help text, and execution storage binding are all known by the command implementation. It is separate from the older bu_opt_desc and bu_opt_desc_meta mechanism; a command uses one mechanism or the other and does not generate duplicate compatibility rows.

The API provides three views of the same schema:

  • bu_cmd_schema_parse binds command-line options to a caller-owned argument structure. bu_cmd_schema_parse_complete additionally requires all positional roles to satisfy their declared cardinality and scalar rules.

  • bu_cmd_schema_describe produces command help, while bu_cmd_schema_describe_json produces machine-readable metadata.

  • bu_cmd_schema_validate performs side-effect-free incremental syntax and scalar-type validation, returning static candidates for completion and highlighting.

The validation result owns its candidate array. Initialize it with BU_CMD_VALIDATE_RESULT_NULL and release it with bu_cmd_validate_result_clear after each use.

Declaring a simple schema

Define an argument structure with the command defaults, then declare options using the BU_CMD_* helpers. Each option names its short spelling, long spelling, storage field, value type, argument label, and help text once.

Use BU_CMD_COUNTING_FLAG for a repeatable no-argument option whose integer storage field must count occurrences (for example, -vv). Ordinary BU_CMD_FLAG options remain boolean even when they are repeated. BU_CMD_CUSTOM_FLAG is the deliberate escape hatch for an order-sensitive no-argument state transition; its parser receives a NULL argument and the selected storage-field address.

BU_CMD_DB_OBJECT and BU_CMD_DB_PATH are compact typed-option forms for database object and path values. They keep libbu application-neutral: an application such as GED may infer its default semantic provider from the value type, while a specialized role may name an explicit provider in its full bu_cmd_option or bu_cmd_operand row.

struct example_args {
    int verbose;
    fastf_t scale;
    const char *output;
};

static const struct bu_cmd_option example_options[] = {
    BU_CMD_FLAG("v", "verbose", struct example_args, verbose,
                "Enable diagnostic output"),
    BU_CMD_NUMBER("s", "scale", struct example_args, scale, "factor",
                  "Scale the result"),
    BU_CMD_FILE("o", "output", struct example_args, output, "file",
                "Write the result to a file"),
    BU_CMD_OPTION_NULL
};

Aliases use BU_CMD_ALIAS_SHORT or BU_CMD_ALIAS_LONG and name the canonical option. Hidden aliases remain accepted by the parser but are omitted from help and completion. Custom values use BU_CMD_CUSTOM; their parser receives a pointer to that option’s storage during execution and a null storage pointer during validation, so custom parsers must treat a null pointer as a side-effect-free syntax check.

Typed options and operands may also use a BU_CMD_*_VALIDATE helper (BU_CMD_OPERAND_VALIDATE for a positional role). Its validator is called without storage during incremental validation; option parsing applies it when binding execution storage, and a command’s positional parser can reuse the same function. This publishes constraints such as an integer range without reducing the value to an opaque custom value.

Canonical keyword values

Use a bu_cmd_value_keyword table when a keyword vocabulary needs stable canonical spellings, accepted aliases, or per-value descriptions. Each row contains a canonical spelling, an optional null-terminated alias list, and an optional description. BU_CMD_KEYWORD_VALUES attaches such a table to a keyword option; BU_CMD_OPERAND_KEYWORD_VALUES does the same for a positional role.

Aliases are accepted during validation and parsing, but static completion offers only the canonical spelling. For a keyword option, parsing also stores the canonical spelling in its const char * field, so execution does not need to duplicate alias normalization. JSON retains the compatible values array of canonical spellings and adds keyword_values, whose entries include canonical, aliases, and description. A string-valued role may publish the same table as completion hints while a validator accepts a wider expression grammar; in that case its original string is intentionally retained.

Native schema lint rejects an option or operand that mixes the legacy simple keyword array with rich values, and rejects empty or duplicate canonical or alias spellings.

Optional and shaped values

Every typed BU_CMD_* option takes one required scalar argument unless it is a flag. Native schemas may instead declare BU_CMD_ARG_OPTIONAL or an explicit bu_cmd_arg_shape using BU_CMD_OPTION_SHAPED. The shape records the token range and one of scalar, token_sequence, comma_list, key_value_list, axis_keyed, range_pattern, or custom. It is emitted in JSON as argument_requirement and argument_shape, so documentation and completion clients do not need a separate mini-language table.

An optional argument consumes zero or one following token; when supplied, it is still checked as the option’s declared value type. This deliberately makes a command such as make_name -s [num] unambiguous: a following word is the optional reset value and is rejected if it is not an integer, rather than silently becoming a positional template.

For a multi-token shape, supply a bu_cmd_value_consume_t consumer. It receives the complete selected token sequence and the address selected by the option’s storage_offset. With a null storage pointer it must perform only side-effect-free validation; with storage it binds the execution value. libbu therefore invokes the same shape parser for execution and incremental validation. A multi-token shape without a consumer is rejected by native schema lint.

A positional role whose value has a one-token mini-language may use BU_CMD_OPERAND_SHAPED. Its shape is published as the operand’s shape in JSON and must describe exactly one token; the operand’s normal cardinality still determines how many such tokens the command accepts. For example, an integer-or-inclusive-range operand can use range_pattern together with a side-effect-free validator, making 100-200 and 100:200 visible as range syntax without misrepresenting the role as a plain integer.

Declarative command relationships

Use a bu_cmd_constraint table for relationships that cannot be represented by an individual option or positional role alone. The compact rows support distinct-option counts (such as mutually exclusive modes) and conditional operand counts (such as either -f file or exactly two object-name operands). Options in a row are canonical names without dashes. BU_CMD_CONSTRAINT_OPTIONS limits the number of selected options; BU_CMD_CONSTRAINT_OPERANDS applies an operand range when any, none, or all of its named options are selected.

Attach the table with BU_CMD_SCHEMA_CONSTRAINTS in the final schema slot:

static const char *mapping_file_option[] = {"f", NULL};
static const struct bu_cmd_constraint rename_constraints[] = {
    BU_CMD_CONSTRAINT_OPERANDS(BU_CMD_CONDITION_ANY_OPTION_PRESENT,
        mapping_file_option, 0, 0, "-f cannot be combined with object names"),
    BU_CMD_CONSTRAINT_OPERANDS(BU_CMD_CONDITION_NO_OPTION_PRESENT,
        mapping_file_option, 2, 2, "source and destination are required"),
    BU_CMD_CONSTRAINT_NULL
};

static const struct bu_cmd_schema rename_schema = {
    "rename", "Rename objects", rename_options, rename_operands,
    BU_CMD_PARSE_OPTIONS_FIRST,
    BU_CMD_SCHEMA_CONSTRAINTS(NULL, rename_constraints)
};

libbu applies these rows to complete execution parsing and complete incremental validation, and publishes them in generated help and JSON. They therefore cannot silently disagree across parsing, validation, and completion.

For a genuinely command-specific side-effect-free syntax rule that cannot be expressed as a constraint row, provide a validator in schema.validation.custom_validate. The callback receives the token array and cursor position and returns a normal bu_cmd_validate_result. It should copy the schema, clear flat.validation.custom_validate, call bu_cmd_schema_validate for the ordinary flat rules, then add only its command-specific rule. bu_cmd_schema_option_present and bu_cmd_schema_operand_count expose the parser’s own alias-, cluster-, shape-, and ---aware interpretation when such a rule depends on a selected canonical option or the number of positional operands; do not duplicate an option scanner in the command. Reserve non-flat languages for an explicit grammar adapter.

Positional roles use BU_CMD_OPERAND. Alongside their scalar type they may carry an opaque semantic-provider identifier. libbu preserves that identifier without depending on any application; GED uses identifiers such as ged.db_object and ged.db_path to supply database-aware validation and completion. A native schema therefore keeps its library-neutral core while still describing application-specific operands precisely.

The compact parser is deliberately for flat option grammars. Commands with subcommands, context-dependent expression languages, or nested commands should publish an explicit syntax adapter instead of forcing that grammar into option rows. Its execution parser preserves argv order for options-first and stop-at-first-operand forms. For a flat interspersed-option command, bu_cmd_schema_parse compacts recognized options and their arguments at the front of the caller-owned argv pointer array, preserving positional operand order in the remaining suffix; execution can therefore consume the same native rows without a second option scanner. A schema-only option for such a command uses BU_CMD_FLAG_UNBOUND; it participates in help, validation, and completion but deliberately refuses bu_cmd_schema_parse, which prevents a metadata table from silently writing through an invented storage offset.

For conventional flat flag sets, the execution parser and validator accept a compact short-option cluster when every member is a no-argument flag: -ah has the same meaning as -a -h. Attached values and clusters containing an option argument remain explicit, avoiding ambiguous getopt-style guessing. An optionless schema treats ordinary leading-dash text as an operand, which keeps raw commands such as echo -literal from inventing an option grammar. The standalone -- marker remains available to make an operand phase explicit.

GED registration

A native command uses GED_DECLARE_COMMAND_SET_WITH_NATIVE_SCHEMA and GED_DECLARE_PLUGIN_MANIFEST_WITH_NATIVE_SCHEMA; the latter publishes the native plugin schema manifest. Command-schema metadata is an internal interface under active development, so a plugin must be rebuilt with the matching libged headers rather than relying on a compatibility adapter.

Once registered, native schemas participate in ged_cmd_schema_json, ged_cmd_validate, ged_cmd_complete_result, ged_cmd_analyze, and ged_cmd_schema_lint. This makes application frontends such as GSH, MGED, and QGED consume the same command metadata without owning separate option tables.

Expression languages

Some commands have a grammar that is not a flat option list. For example, the parser-owned find-style vocabulary of db_search is available through db_search_syntax_terms in rt/search.h. Its terms report argument kinds such as ordinary strings, integers, primitive types, and variadic -exec commands. Completion clients should use that exported view rather than copy the db_search parser table. db_search_syntax_plan_start and db_search_syntax_exec_substitution expose the parser’s boundary and {} substitution rules for the same reason.

GED registers such a command with a ged_cmd_grammar adapter rather than inventing flat option rows. The adapter supplies cursor-aware validation, whole-line token analysis, JSON description, and lint hooks; the command’s ordinary GED execution callback remains responsible for execution. Its JSON uses kind: grammar_adapter and must describe parser-owned terms from the same vocabulary used by execution. Consequently GSH, MGED, and QGED invoke the same adapter through the normal GED completion and analysis APIs without special-casing a command name.
