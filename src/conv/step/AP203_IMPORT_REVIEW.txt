--------------------------------------------------------------------------------
 1. KEY ARCHITECTURAL FINDINGS
--------------------------------------------------------------------------------

 * The step-g importer is compiled with -DAP203 against
     src/conv/step/step-g/ap203.exp
   whose schema header is:
     SCHEMA config_control_design;
   i.e. it targets *classic AP203 (edition 1)*, the CONFIG_CONTROL_DESIGN
   MIM.  That schema defines 254 ENTITY types.

 * classic AP203 has NO tessellated/mesh entities.  There is no
   tessellated_shape_representation / triangulated_face /
   complex_triangulated_surface_set etc. (those were introduced later in
   AP203e2 / AP242).  The ONLY mesh-like geometry available in AP203 is:
       faceted_brep      (a manifold_solid_brep whose faces are planar)
       poly_loop         (an ordered vertex polygon face bound)
   Before this pass neither was handled, so faceted/polyhedral solids were
   dropped silently.  Adding them is the single largest "missing geometry"
   win available inside the AP203 schema and is the flagship change here.

 * Import pipeline (STEPWrapper::convert):
     - Walks all instances; keys off top-level representation entities:
         shape_definition_representation, advanced_brep_shape_representation,
         manifold_surface_shape_representation, shell_based_surface_model,
         geometric_set, geometrically_bounded_surface_shape_representation,
         shape_representation_relationship, context_dependent_shape_representation.
     - AdvancedBrep* -> ON_Brep (OpenNURBS) -> BRL-CAD "brep" primitive.
     - ShellBasedSurfaceModel -> plate-mode ON_Brep.
     - Assembly hierarchy from next_assembly_usage_occurrence /
       context_dependent_shape_representation (as combs w/ transforms).
     - Entities self-register a factory method (Factory::RegisterClass) and
       implement LoadONBrep()/LoadONCurve()/LoadONSurface() for geometry.

 * Geometry conversion status was audited for every one of the 254 schema
   entities (see the table in section 5).  Summary counts:
       FULL     :  50   geometry/data faithfully imported
       PARTIAL  :   9   imported but lossy / limited
       STUB     :  28   class exists but conversion is a no-op / error
       MISSING  : 101   no class at all - silently dropped
       METADATA :  65   non-geometric admin/PDM entity (acceptable to skip)
   (253 rows reported; "effectivity" listed once.)
   Most MISSING entities are administrative/PDM (person, approval, date,
   contract, effectivity, ...) which carry no geometry and are correctly
   ignored.  The geometry-bearing gaps are enumerated in the TODO (section 4).

--------------------------------------------------------------------------------

 Build (Windows, Ninja tree):
     cmake --build out/build/x64-Release --target step-g
 (from a VS x64 developer environment; see .claude memory "build-on-windows").

   bounding box 10 x 10 x 10 mm (volume 1000 mm^3).

 3. USER-VISIBLE CHANGES (this pass)   [x] done  [ ] planned
--------------------------------------------------------------------------------

 [x] NEW: Faceted / mesh solid import.  faceted_brep + poly_loop +
     faceted_brep_shape_representation are now imported as a BRL-CAD BoT
     (bag of triangles) solid, wrapped in a region and attached to the
     product hierarchy.  Previously these were silently dropped.
       - Planar polygon faces (poly_loop) are fan-triangulated.
       - Shared vertices are welded (deduplicated by cartesian_point id).
       - Units are honored (coordinates scaled by the shape rep's length
         conversion factor); output BoT is RT_BOT_SOLID / RT_BOT_UNORIENTED
         so ray tracing does not depend on STEP facet winding.
       New files:
         step-g/PolyLoop.{h,cpp}
         step-g/FacetedBrep.{h,cpp}
         step-g/FacetedBrepShapeRepresentation.{h,cpp}
       New API:
         BRLCADWrapper::WriteBoT()
       Touched:
         step-g/ConnectedFaceSet.h (Faces() accessor)
         step-g/Face.h             (Bounds() accessor)
         step-g/FaceBound.h        (GetBound() accessor)
         STEPWrapper.cpp           (convert_WriteBoT + convert() wiring)
         step-g/CMakeLists.txt     (new sources/headers)

 [x] FIX: Bezier curve import.  bezier_curve previously OVERRODE the working
     B-spline handler with a "not implemented" stub, so every bezier_curve
     edge was dropped.  It now delegates to BSplineCurve::LoadONBrep (a Bezier
     is a B-spline with an implicitly clamped knot vector, which that code
     already generates correctly).

 [x] FIX: Rational Bezier curve import.  Same fix; delegates to
     RationalBSplineCurve::LoadONBrep.

 [x] FIX: Bezier surface import.  bezier_surface was a stub; now delegates to
     BSplineSurface::LoadONBrep.

 [x] FIX: Rational Bezier surface import.  Was a stub; now delegates to
     RationalBSplineSurface::LoadONBrep.

 [x] NEW: Assembly instancing via mapped_item + representation_map.  A
     mapped_item instances the geometry of a representation_map's
     mapped_representation, transformed so the map's mapping_origin coincides
     with the item's mapping_target.  These are now realized as comb
     memberships (with the placement matrix) of the containing
     representation's object, so a component defined once can be placed many
     times (e.g. four wheels from one wheel body).  Previously all mapped_item
     instances were dropped.
       New files:
         step-g/MappedItem.{h,cpp}
         step-g/RepresentationMap.{h,cpp}
       Touched:
         STEPWrapper.cpp  (assembly_placement_matrix() helper + a mapped_item
                           resolution pass; faceted naming pass now also
                           records shape-rep -> object name so faceted
                           components can be instanced)
         step-g/CMakeLists.txt  (new sources/headers)
     Validated: src/conv/step/test_mapped_assembly.stp - one 10mm faceted
     cube instanced 3x at (0,0,0),(30,0,0),(0,40,0) yields an "assembly" comb
     with 3 correctly-placed "widget" members and an exact 40x50x10 mm bbox.

 [x] FIX: Plate-mode and geometrically-bounded geometry now honor their
     container's axis placement.  convert_WritePlateBrep()/convert_WriteBSpline()
     previously ignored the shape representation's axis2_placement_3d - a dead
     '#if 0' block there referenced a non-existent 'aBrep' variable copy-pasted
     from convert_WriteBrep, so it never compiled (which is why it was disabled
     rather than fixed).  Both functions now take an Axis2Placement3D* and apply
     it via assembly_placement_matrix(axis, NULL), sourced from the containing
     ManifoldSurfaceShapeRepresentation (plate) or
     GeometricallyBoundedSurfaceShapeRepresentation (bspline;
     GetAxis2Placement3d() added to that class to match the other shape reps).
     Identity placement when the representation has no axis, so files without
     one are unchanged.
     Validated: src/conv/step/test_gbss_axis.stp - a geometrically_bounded
     surface rep whose container axis origin is (100,0,0) with a flat 10x10
     bilinear B-spline patch.  The imported patch region carries member matrix
     [100,0,0] (region center 105,5,0), i.e. the container axis is applied;
     with the old code it would sit at 0..10.  Full pre-existing suite
     (faceted tetra, mapped assembly, cylcub BoTs, block brep) still passes
     (identity axes -> unchanged), confirming no regression.

 Behavioral notes / known limitations of the faceted path:
   - Fan triangulation is exact for convex faces (the common case, since
     exporters usually emit triangles/convex quads).  Non-convex planar
     faces should use ear-clipping (libbg) -- see TODO.
   - Faceted solids are linked to products through
     shape_definition_representation.  Faceted geometry reached only through
     shape_representation_relationship is still imported as a top-level
     region but not nested under a product comb -- see TODO.

       instantiated; downgrade to debug-level messages.

       out of scope for the AP203 importer but noted for completeness.

       path to more surface/curve element types.
